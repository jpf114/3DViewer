# 量测成果管理设计

## 目标

在不引入新复杂面板的前提下，把当前“量测只是图层”的状态推进到“量测成果可管理”的第一阶段。第一阶段聚焦三个结果：

1. 用户可以在图层树中重命名量测成果
2. 量测成果在再次编辑后，不会被自动摘要名称覆盖掉自定义名称
3. 自定义名称可继续通过现有工程文件与图层持久化机制保存和恢复

## 现状

当前量测成果已经支持：

- 创建
- 再编辑
- 导出 GeoJSON
- 删除
- 工程保存/打开恢复

但仍有明显短板：

- 用户不能正式重命名量测成果
- 编辑量测后会重新生成“距离/面积摘要名称”，这会覆盖成果语义名称
- 成果管理入口分散，缺少统一的管理体验

## 方案对比

### 方案 A：新增独立“量测成果面板”

优点：
- 最完整，后续可扩展排序、筛选、批量操作

缺点：
- 改动面广，UI 和状态同步成本高
- 当前阶段投入偏大

### 方案 B：在现有图层树上补齐成果管理

优点：
- 成本低，复用现有图层树和菜单结构
- 对当前代码最友好
- 能快速补齐重命名这个最缺的环节

缺点：
- 仍不是完整成果工作台

### 方案 C：只在属性面板增加名称输入框

优点：
- 改动小

缺点：
- 操作路径绕，图层树和属性面板双向同步会更别扭

## 结论

采用 **方案 B**。

第一阶段在 `LayerTreeDock -> MainWindow -> ApplicationController -> Layer` 这条链路上增加“重命名”能力，并修复量测再次编辑时的命名覆盖问题。这样能把“成果管理”的核心缺口补上，同时不把 UI 做得过重。

## 设计

### 1. 图层树重命名入口

在 `LayerTreeDock` 的右键菜单中增加“重命名”动作，并允许当前项进入原地编辑。编辑完成后发出新信号，把 `layerId` 和新名称传给上层。

这项能力不只服务量测图层，也适用于普通图层，但当前主收益是量测成果。

### 2. MainWindow 信号透传

`MainWindow` 增加 `layerRenameRequested(const QString &, const QString &)`，将图层树的重命名事件透传给 `ApplicationController`。

### 3. ApplicationController 更新图层名称

控制器负责：

- 校验图层是否存在
- 忽略空名称
- 去掉首尾空白
- 将名称写回 `Layer`
- 更新当前详情面板

### 4. 量测编辑后的命名策略

当前量测再次编辑时，系统会重新生成摘要名称。新策略改为：

- 如果图层当前名称仍等于“旧的自动摘要名称”，说明用户没有自定义名称，则继续更新为“新的自动摘要名称”
- 如果图层当前名称已经不是旧摘要名称，说明用户自定义过名称，则保留当前名称，不再覆盖

这样既保留自动命名的便利，也保护人工命名成果。

## 测试

新增或扩展冒烟测试覆盖：

1. 图层树支持发出重命名请求
2. 控制器收到重命名后会更新图层名称
3. 自定义量测名称在再次编辑后仍被保留
4. 自定义量测名称通过工程保存/打开后仍被恢复
