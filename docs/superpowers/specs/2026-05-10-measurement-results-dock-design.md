# 量测成果工作台设计

## 目标

在现有图层树之外，新增一个独立的“量测成果工作台”停靠面板，用于承载量测成果列表、批量操作和统一入口，让当前量测能力从“工具可用”推进到“成果可管理”。

本阶段目标聚焦于一个最小可用版本：

1. 用户能在独立面板中看到全部量测成果
2. 用户能在面板中单选/多选量测成果
3. 用户能通过面板统一执行定位、编辑、删除、批量删除、批量导出
4. 面板与当前 `LayerManager` / `MainWindow` / `ApplicationController` 状态保持一致

## 背景

当前项目已经具备：

- 量测成果创建
- 量测成果再次编辑
- 单条量测导出 GeoJSON
- 单条量测删除
- 清空全部量测成果
- 工程保存 / 打开恢复量测成果
- 图层树中重命名量测成果

但当前量测成果管理仍有明显问题：

- 成果入口分散在工具栏、右键菜单、量测菜单中
- 图层树是场景组织结构，不适合承担完整成果工作流
- 批量操作缺少可视化列表与多选基础
- 后续如果接入通视、可视域、高程剖面等分析成果，现有入口无法自然承载

## 设计原则

### 1. 图层树与成果工作台分责

图层树继续负责：

- 图层顺序
- 图层显隐
- 场景组织

量测成果工作台负责：

- 成果列表
- 当前成果上下文
- 批量操作
- 成果统一入口

这样符合桌面 GIS 工具中“场景管理”和“成果管理”分层的常见做法。

### 2. 不引入第二套状态源

工作台不是新的数据模型中心，真实状态仍然来自：

- `LayerManager`：量测图层与属性
- `ApplicationController`：动作分发、导出、删除、选择联动

工作台只负责显示与发信号，不持有独立业务真相，避免状态分叉。

### 3. 先做量测，接口为成果中心预留

第一版只显示 `Measurement` 图层，但命名与接口边界尽量保持可扩展，后续能够自然接入更多分析成果，而不需要推翻 UI 结构。

## 方案对比

### 方案 A：继续复用图层树

优点：

- 改动最小

缺点：

- 图层树语义不纯
- 多选和批量操作体验差
- 无法自然扩展为成果工作台

### 方案 B：新增独立量测成果工作台

优点：

- 职责清晰
- 操作路径统一
- 后续可自然扩展到更复杂成果体系

缺点：

- 需要新增一个 Dock 和同步逻辑

### 方案 C：直接做通用成果中心

优点：

- 一步到位

缺点：

- 当前需求和代码基础还不支持一次做大
- 容易把本阶段拖成平台化改造

## 结论

采用 **方案 B**。

第一版实现一个独立 `MeasurementResultsDock`，只管理量测成果；同时在命名、信号和控制器入口上保留向“统一成果中心”过渡的空间。

## 信息结构

工作台采用表格或列表形式展示量测成果，每条成果展示：

- 名称
- 类型（测距 / 测面）
- 摘要值（长度或面积）

第一版不展示时间戳、创建来源、标签、分组。

列表支持：

- 单选
- 多选
- 当前项高亮

## 交互设计

### 1. 基础选择联动

当用户在工作台中选中一条量测成果时：

- `MainWindow` 当前图层切换到对应量测图层
- 属性面板展示该量测成果详情

当用户在图层树或其他入口选中量测成果时：

- 工作台同步高亮对应条目

### 2. 操作入口

工作台提供统一操作按钮或右键菜单，至少支持：

- 定位到成果
- 编辑当前成果
- 删除当前成果
- 批量删除
- 批量导出

其中：

- “编辑当前成果”只对单选有效
- “定位到成果”对单选优先，若多选则默认定位第一条
- “批量删除”和“批量导出”只对多选或非空选区有效

### 3. 批量导出策略

第一版不做复杂打包格式，采用保守策略：

- 逐条导出为多个 GeoJSON 文件
- 用户先选择导出目录
- 文件名基于量测名称做安全清洗

如果出现重名文件，控制器负责生成稳定的去重文件名。

## 架构设计

### 1. 新增 UI 组件

新增 `MeasurementResultsDock`：

- 负责渲染量测成果列表
- 负责发出选择、编辑、定位、删除、批量删除、批量导出等信号
- 提供刷新、增量新增、更新、删除、选中同步等接口

### 2. MainWindow 负责挂载与透传

`MainWindow` 负责：

- 创建并挂载 `MeasurementResultsDock`
- 将工作台信号透传给 `ApplicationController`
- 在图层新增、删除、重命名、详情切换时同步工作台显示

### 3. ApplicationController 负责动作落地

`ApplicationController` 负责：

- 从 `LayerManager` 收集量测图层并驱动工作台刷新
- 响应工作台发起的选择、编辑、定位、删除、批量删除、批量导出
- 复用现有单条编辑 / 导出 / 删除逻辑
- 补充批量导出逻辑

### 4. 数据表示

第一版不新增持久化结构体，工作台所需字段直接由量测图层推导：

- `layer.id`
- `layer.name`
- `measurementData.kind`
- `measurementData.lengthMeters`
- `measurementData.areaSquareMeters`

必要时可在 UI 层引入一个轻量 view-model 结构，用于列表展示，但不成为新的业务数据源。

## 错误处理

第一版保持与现有桌面工具风格一致：

- 无选中时操作，状态栏给出提示
- 批量导出目录未选择时，不执行导出
- 某条量测数据不完整时，跳过并提示失败数量
- 批量删除中若某条不存在，按幂等方式忽略

## 测试策略

新增或扩展冒烟测试，至少覆盖：

1. 工作台存在并能显示量测成果列表
2. 工作台选中项能联动主窗口当前量测
3. 单条删除只删除对应量测图层
4. 批量删除只删除选中的量测图层
5. 批量删除不会影响非量测图层
6. 批量导出动作至少覆盖控制器分发和基础成功路径

如果实现中引入纯函数（例如导出文件名去重策略），优先补单元测试而不是把所有逻辑都塞进冒烟测试。

## 本阶段明确不做

第一版不包含：

- 搜索 / 筛选
- 排序配置
- 时间戳
- 成果分组
- 非量测分析成果混合展示
- 复杂导出包格式

这些内容留到“统一成果中心”阶段再处理。

## 落地顺序建议

1. 新建 `MeasurementResultsDock` 和最小列表展示
2. 接通 `MainWindow` 挂载与选择联动
3. 接通删除 / 批量删除
4. 接通单条编辑 / 定位
5. 接通批量导出
6. 补测试与文档

## 成功标准

当以下条件同时满足时，认为第一版达成目标：

1. 用户能在独立工作台看到所有量测成果
2. 用户能用多选完成批量删除
3. 用户能从统一入口批量导出量测成果
4. 工作台与图层树选择保持一致
5. Debug 回归测试通过
