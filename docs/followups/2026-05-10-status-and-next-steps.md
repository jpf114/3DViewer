## 2. 当前项目现状

### 2.1 工程状态

Windows 开发与发布链路已打通：

- `Debug` 配置成功
- `Debug` 编译成功
- `Debug` 测试通过（`13/13`）
- `Release` 配置成功
- `Release` 编译成功
- `Release` 测试通过（`13/13`）
- `Release` 已安装到 `install/Release`
- 安装目录中的 `3DViewer.exe` 已验证可启动并稳定存活，不是秒退

这意味着项目已经从“环境可能有问题、构建不稳定”的阶段，进入“可稳定开发、可稳定回归、可稳定打包”的阶段。

### 2.2 产品状态

当前项目已经具备一个桌面 3D GIS MVP+ 的核心骨架：

- Qt 主窗口、图层树、属性面板、状态栏
- osgEarth/OSG 三维地球场景
- 底图配置与切换
- 影像 / 高程 / 矢量导入
- 图层显隐、顺序、透明度、基础持久化
- 拾取与属性展示
- 基础距离 / 面积量测
- 模型图层基础接入

当前项目真实定位应当是：

> 一个已具备稳定工程基础、具备基础 GIS 浏览与交互能力的桌面 3D GIS 原型产品。

## 4. 当前仍然存在的短板

### 4.1 三维内容能力仍偏早期

虽然模型图层主线已经接入，但整体三维内容能力仍弱于成熟 3D GIS 产品：

- 模型格式支持仍需继续增强
- 倾斜摄影 / 3D Tiles / 点云仍未进入主线
- 三维专业分析能力尚未展开

### 4.2 量测还不是完整成果体系

当前量测更像工具，而不是成果管理能力。仍需补齐：

- 量测结果统一管理面板
- 批量操作与统一列表
- 更完整的导出与复用
- 后续分析成果的统一管理模型

### 4.3 工程文件主线已落地，但仍有第二阶段短板

当前已具备：

- `.3dproj` 工程文件
- 当前底图保存 / 恢复
- 当前相机视角保存 / 恢复
- 普通图层保存 / 恢复
- 模型位置保存 / 恢复
- 量测成果保存 / 恢复

但仍有后续增强空间：

- 工程内图层样式与更多渲染参数
- 最近工程与未保存变更提示
- 工程内资源健康检查（丢失路径、失效数据源）
- 工程版本升级兼容策略

### 4.4 表达能力仍偏 MVP

未来仍需增强：

- 栅格渲染增强
- 矢量符号化
- DEM 表达增强
- 更明确的导入 / 错误 / 加载反馈

## 5. 后续总体计划

后续不再平均用力，而按阶段推进。

### 阶段一：产品闭环补强

目标：把当前 MVP 做成“更像正式工具”的版本。

优先项：

1. 量测成果化与管理增强
2. 模型能力增强
3. 工程文件体验增强

### 阶段二：GIS 表达增强

目标：把“能显示”提升为“更专业地显示”。

优先项：

1. 栅格增强（伪彩、拉伸、单波段等）
2. 矢量样式增强（分类、分级、标签）
3. 交互反馈增强（错误、加载、状态）

### 阶段三：三维 GIS 专业能力拓展

目标：进入更成熟的三维 GIS 能力层。

优先项：

1. 通视分析
2. 可视域分析
3. 高程剖面
4. 3D Tiles / 倾斜摄影
5. 点云支持

---

## 6. 当前推荐优先级

按照现实收益与阶段匹配度，推荐近期执行顺序如下：

1. 量测成果管理
2. 模型能力增强
3. 工程文件体验增强
4. 栅格 / 矢量表达增强

## 7. 下一阶段明确主线

### 主线：量测成果管理

“工程保存 / 打开”第一阶段已完成，下一主线切换为“量测成果管理”。

原因：

- 工程文件能力已经具备闭环，继续深挖边际收益下降
- 量测当前已经能保存恢复，但仍缺统一列表、重命名、删除与成果管理体验
- 这条线能直接增强 GIS 工具属性，比继续停留在底层工程文件细节上更划算
- 完成后可自然衔接更复杂的分析成果管理

### 下一阶段设计必须考虑的内容

量测成果主线应重点考虑：

- 独立成果列表
- 重命名
- 删除
- 二次编辑入口统一
- 导出入口统一
- 后续分析成果的复用接口

### 下一阶段产出目标

后续围绕“量测成果管理”推进时，应至少形成：

1. 设计文档
2. 实现计划
3. 最小闭环实现
4. 对应测试覆盖

---

## 8. 对后续开发的约束

为避免后续偏航，继续遵守以下原则：

1. 先补能力闭环，再开新大类能力
2. 所有新增功能应尽量带最小测试覆盖
3. 不将 `Chart` / `Scientific` 对外当作正式可用能力宣传，直到其完整接入
4. 保持“应用层状态”和“渲染场景状态”分层，不把 UI 直接绑死到渲染逻辑
